# Implementation Plans

Implementation plans: **how** the task will be executed.

## When to create a file here

- After the spec is approved and before coding begins
- One file per task, named to match the corresponding spec file

## Required contents (per CLAUDE.md)

- `intended approach` — the technical strategy
- `affected files / systems` — specific files and modules touched
- `execution steps` — ordered list of changes to make
- `validation steps` — automated and manual checks to run
- `key assumptions` — things assumed true that may need verification
- `known risks` — what could go wrong
- `decision points requiring escalation` — conditions that should pause execution

## Rules

- Plans explain **how**, not **what** (the spec owns what/why)
- Plans must include validation so executors don't improvise it
- Plans must flag what is core vs. optional vs. escalation-required
- If reality diverges from the plan, document it — do not silently fix
